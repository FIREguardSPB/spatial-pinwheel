# AUDIT FIX81

## Корневая проблема до фикса
По HAR и скриншотам основной сбой происходил не из-за одного bad endpoint, а из-за фронтовой схемы загрузки:
- слишком много параллельных GET,
- часть из них периодически повторялась,
- страницы состояли из большого числа независимых query,
- запросы застревали в browser/proxy queue,
- затем фронт сам обрывал их по таймауту,
- UI интерпретировал это как массовую деградацию backend.

## Что исправлено архитектурно
Исправление выполнено не точечными патчами, а через coordinator-first модель:
- backend теперь умеет отдавать page bootstrap payload;
- frontend страниц опирается на эти payload, а не на десятки мелких источников;
- сетевой граф на страницу стал намного проще;
- вероятность забить очередь браузера должна существенно снизиться.

## Остаточные риски
- если сам backend-обработчик page-bootstrap endpoint зависает, соответствующая страница всё равно будет деградировать;
- backtest и tokens пока не переведены на новый coordinator pattern, но они не были основным источником сетевого шторма;
- график по свечам всё ещё отдельный и зависит от `/candles`, что является нормальным исключением.

## Критерий успеха
После деплоя FIX81 в DevTools/Network на страницах `Настройки`, `Сигналы`, `События`, `Счёт`, `Сделки` должен быть один основной `GET /api/v1/ui/...` на страницу вместо каскада из множества `status/runtime/settings/watchlist/...` запросов.
